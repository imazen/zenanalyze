### 2.1 The `partition_range` debug story

Three separate sweep runs (v04-full, v05, v05b) got zero AVIF chunks because every cell panicked at `zenravif-0.1.3/src/av1encoder.rs:1559: bad size N`. `N` matched the first element of the `partition_range` tuple (0, 1, 3 for the values we passed).

**Cause:** rav1e's `min_partition_size` field expects an AV1 BLOCK_SIZE log2 value where valid range is {4..7} (corresponding to 16×16 .. 128×128 partitions). Values {0, 1, 3} are below the valid floor; rav1e panics rather than returns an error.

**Fix in v05c:** dropped `partition_range` from the knob grid entirely. The remaining knobs (speed, complex_prediction_modes) are sufficient for picker training and don't trigger the panic.

**Follow-up not in this session:** zen-metrics CLI should validate `partition_range` values against rav1e's spec OR rav1e/zenravif should return `Result` instead of panicking. Either fix prevents an entire fleet from silently producing zero AVIF output.

### 2.2 Chunking strategy

Smaller chunks = lower tail latency. v05c chunks: 4 imgs × (1 speed | effort) × full q-grid = 76–96 cells/chunk, ~5–15 min wall on a 12-core box. Total chunks: 5598 (avif 2488 + jxl 3110).

**Lesson:** if chunks are >1 hour wall, kill + redesign. v05/v05b tried 12-img × full-knob-grid chunks — single chunks took 12 hours of wall time.

## 3. Picker training pipeline (v0.5)

End-to-end: **sweep → adapt → extract features → train → bake → A/B verdict**.

### 3.1 Sweep TSV → zentrain schema

`zenmetrics_sweep_adapter.py` translates v0.3.0+ sweep TSVs to the v0.1 zentrain schema:
```
image_path | size_class | width | height | config_id | config_name | <axis cols> | bytes | zensim | encode_ms | decode_ms
```

For zenavif: `config_name = s{speed}_q{q}` (no tune axis post-v04 spec drop).
For zenjxl: `config_name = e{effort}_d{distance}_q{q}` with `q=75` dummy.

Pre-existing bug: trainer expects `score_ssim2` column; v05c sweep TSVs have `score_ssim2_gpu`. Fix: pass `--metric zensim` explicitly to the adapter.

Pre-existing TSV corruption issue: 1 row in 281,486 had a partial knob_tuple_json (likely a chunk upload race). Filtered with a csv.reader-aware cleaner; lost 1 row of 281,485 valid rows.

### 3.2 Feature TSV generation

`extract_features_natural` (zenavif example, codec-agnostic): runs `zenanalyze::analyze_features_rgb8` over a manifest of (relative_path, bytes, width, height, axis_class, source, description). Output: ~95 features per image including the 51 `feat_*` axes used by current pickers.

For 1241 imgs: ~5 seconds wall, no failures.

**Path-flattening invariant:** The adapter writes `image_path = {sub}__{file.png}` (single-underscore replacement of `/`). For variants the prefix is `variants__{sub}__{file.png}`. The features TSV must match — use the post-extraction script in this session that strips the `mlp-tune__` prefix and replaces `mlp-tune-variants__` with `variants__`.

### 3.3 Training (`train_hybrid.py`)

Hybrid-heads picker: per-cell HistGradientBoostingRegressor teachers + a unified MLP student.

| Codec | Cells | Output dim | MLP | Params | Val argmin acc | Val mean overhead | Train→val gap |
|---|---|---|---|---|---|---|---|
| zenavif v0.5 | 4 (speed3/5/7/9) | 8 | 112→128→128→8 | 32008 | **72.6%** | 5.49% | +4.62 pp |
| zenjxl v0.5 | 5 (effort1/3/5/7/9) | 15 | 112→128→128→15 | 32911 | **51.0%** | 8.81% | +6.00 pp |

Safety violations fired (informational, didn't block bake):
- OVERFIT (both codecs)
- PER_ZQ_TAIL @ p99 (synthetic edge cases — `thin_lines_sp8_512x512.png` etc dominate)
- PER_SIZE_TAIL @ medium p99
- DATA_STARVED_SIZE (42 cells with <50 train rows; mostly tiny/zq combos that don't exist in our corpus)
- UNCAPPED_ZQ_GRID (per-image zensim ceilings would help but we don't compute them yet)

### 3.4 Bake (`bake_picker.py` → `zenpredict-bake`)

i8 quantization, ZNPR v3 binary format. Per-codec output: ~40 KB .bin + ~80 KB manifest.json.

**Patch needed:** the trainer emits `output_specs` with `transform: "discrete_set"` for some scalars; the Rust `zenpredict-bake` only accepts `{identity, sigmoid, sigmoid_scaled, exp, round}`. Fix: a small JSON patch step replaces `discrete_set` → `round` before calling `bake_picker.py`. (Tracked separately; trainer should emit `round` directly.)

### 3.5 Held-out A/B (table-lookup)

`tools/holdout_ab_lookup.py` (zq-banded) for zenavif. `tools/holdout_ab_lookup_jxl.py` (distance-banded, NEW this session) for zenjxl. Holdout: 20% of images, deterministic seed=7.

The harness:
1. Loads the hybrid_heads_manifest from the JSON model.
2. Runs the MLP forward pass in numpy (300-d feat vector + engineered xe; matches `train_hybrid.py:945-985`).
3. Picks `argmin(out[bytes_log_range])` as the cell choice.
4. Looks up the actual sweep result for `(image, picker_cell, q-or-distance)` and `(image, default_cell, q-or-distance)`.
5. Reports per-band Δbytes %, Δzensim_pp, win rate (Δbytes < −0.1%).

Verdict gate (configurable, default in the script): **SHIP** if `mean(Δbytes) < −0.5%` AND `mean(Δzensim_pp) > −0.5`.

### 3.7 zenjxl v0.5 results (HOLD with diagnosis)

vs `effort=7`:

| Band (distance) | n | Mean Δbytes | Win rate | Δzensim_pp |
|---|---:|---:|---:|---:|
| tight (0.05..1.0) | 1717 | −0.70% | 43.1% | +0.15 |

**Next steps to ship zenjxl picker:**
1. Add `encode_ms` to the picker loss → effort=3 wins on speed, picker should reflect that cost.
2. More compressibility-prediction features (orientation/edge-density variants) — current 51-feat set may be insufficient.
3. Investigate corpus balance — synthetic + screen content may bias toward effort=3.

## 4. Per-content-class quant tables (GLASSA SA) — definitive NO-SHIP

**Hypothesis:** screens / documents / line-art have fundamentally different DCT frequency distributions than photos; a photo-tuned table over-allocates bits to high-freq AC that screens don't have.

**Two probes run, both with real measurement (no SA optimization needed to falsify):**

### B (preserve high-freq more — flatten high-freq quant 2× lower)
- Screens at-q ΔBA: −0.12 to −0.35
- Photos at-q ΔBA: −0.03 to −0.11
- Bytes: +37–115% screens, +69–283% photos (paid huge bytes for tiny BA gain)
- **MIXED — directional signal exists (screens 3× more sensitive than photos) but at-bytes-matched the perturbation lost on both classes.**

### B′ (the corrected direction — coarsen high-freq quant 2×–8×)
- Screens at-q ΔBA: **+0.757** at q=85 (huge degradation, opposite of hypothesis)
- Photos at-q ΔBA: +0.264 at q=85
- Bytes shrink: 5.2% screens, 2.2% photos
- **WEAK — screens degrade 3× more under HF coarsening, opposite of hypothesis. Sharp UI/text edges carry HF AC that butteraugli weights heavily; they are MORE sensitive to HF coarsening than photos, not less.**

**36 of 36 (cluster × variant × q) cells fail the +3.0 pareto ship gate.**

### Conclusion

The per-content-class quant table hypothesis is dead at the high-frequency-band axis. v4 photo tables are the universal default. Documented in:
- `imazen/zenjpeg#138` — NO-SHIP finding
- `imazen/zenjpeg#139` — probe binary
- `imazen/zenanalyze#71` — sa_piecewise_v5 validation report
- `imazen/zenanalyze#72` — per_class_signal_probe_bprime_2026-05-04.md

### Axes NOT falsified by these probes (future work, only if motivated)

1. Different metric (zensim2 / fast-ssim2 weight frequencies differently than butteraugli)
2. Different perturbation axis (mid-frequency band rows 2-5 × cols 2-5)
3. Different cluster cut (intrinsic content features vs source-corpus subdir labels)
4. Different chroma subsampling regime (4:4:4 vs 4:2:0 may hide or expose different per-class signal)

Each is a 30-min cheap probe before any SA budget.

### Pre-existing v4 photo tables remain optimal

v4 (the production ship): trained on CID22-photo with butteraugli-GPU SA. Mean train pareto +6.602; OOS proxy +6.385 (96.7% of train). Theoretical refit headroom +0.218 << +3 ship threshold.

The local SA optimizer (`~/work/glassa/examples/optimize_jpegli_tables.rs`) is bitrotted (9 errors against fast-ssim2 0.8 + jpegli-rs 0.12 + Rust 2024 edition). Not on critical path until evidence justifies a v5 refit.

## 5. zen-metrics 0.4.0/0.5.0 infrastructure

### 0.4.0 — feature parquet sidecar

`--feature-output <path.parquet>` flag on the sweep subcommand. Per-cell row schema:
```
image_path | codec | q | knob_tuple_json | zensim_score | feat_0 | feat_1 | ... | feat_299
```

305 columns total. Compression: zstd-3, 256-row Arrow batches. Per-chunk file (`features-${chunk_id}.parquet`).

**Implementation:** zensim 0.2.8 added `Zensim::compute_extended_features() -> (score, Vec<f32>)`. zen-metrics-cli wraps it via `crates/zen-metrics-cli/src/sweep/feature_writer.rs` (Apache Arrow's `parquet` crate).

**End-to-end test (committed):** 6 cells × 2 metrics, parquet ↔ TSV inner-join clean, zensim scores match exactly.

**On-disk size:** ~60 KB compressed parquet for a 6-cell test (305 cols × 6 rows zstd-3 baseline overhead). At sweep scale (1M cells): ~50–80 MB compressed. Trivial vs encode wall-time.

### 0.4.1 — features-backfill subcommand (PR #1)

`zen-metrics features-backfill --input-tsv <chunk.tsv> --output-parquet <chunk.parquet> --corpus-root <dir>` (local mode).
`zen-metrics features-backfill --run-id <RUN_ID> --codec <codec> --corpus-root <dir>` (R2 mode — walks `s3://zentrain/<run-id>/<codec>/*.tsv`).

Idempotence:
- Pre-skip: existence check on the target parquet/key.
- Atomic local write: `<output>.parquet.tmp` → `rename()` to final.
- Atomic R2 upload: `<key>.uploading` → server-side `mv` to final.
- TSVs are read-only; never rewritten.

Image-path resolution: 4-level fallback covering as-written, flat-corpus, onstart_v3 stage flattening (`__` → `/`), and recursive basename walk.

5 integration tests + 8 unit tests, all green; clippy clean.

### 0.5.0 — butteraugli single-compute multi-column (PR #2)

`run_metric` returns `Vec<(&'static str, f64)>`. Single butteraugli `compute()` produces both columns:
- CPU: `score_butteraugli_max` + `score_butteraugli_pnorm3`
- GPU: `score_butteraugli_max_gpu` + `score_butteraugli_pnorm3_gpu`

Drops `MetricKind::ButteraugliMax` and `ButteraugliMaxGpu` enum variants (no longer needed; the `butteraugli` flag now emits both norms).

Sweep TSV header expansion + `score`/`compare`/`batch` subcommand outputs all updated. 25 tests pass.

**Coordination caveat:** PRs #1 and #2 both touch `main.rs` and `sweep/run.rs`. Author used non-overlapping regions; merge conflict expected to be trivial. Resolve PR #1 first.

5. **Cost**: $0 volunteer / ~$4k Prolific to 500k pairs (saturation).
6. **Per-class IS a real metric-side signal** even though it doesn't justify per-class encoder tables — should be an MLP input (one-hot or learned embedding), not a per-class table choice.

1. Land V0_4..V0_7 on `main` (existing branch work — separate from squintly-collected data).

## 9. Methodology notes (for future sessions)

### What worked

- **Per-host shuffled chunks list** to reduce duplicate-work collisions (atomic claim is better, but shuffle alone gets ~80-90% efficiency).
- **R2-binary override** (`SWEEP_BIN_OVERRIDE=s3://...`) — uploads new binary in seconds, vast.ai pulls in seconds. Far better than rebuilding from source on each box (10-15 min).
- **Static-binary onstart** (s5cmd + jq + mc) — boots in 30s instead of 3-8 min apt-get.
- **Cheap signal probes before SA budget** — both GLASSA per-class probes (B and B′) ran in ~30 min of compute and definitively answered the gating question without spending hours on SA.
- **Distance-banded A/B for zenjxl** — caught a real harness limitation that produced a misleading first verdict; the corrected harness gave a real measurement.

- **Sub-agents under-perform the main agent on multi-hour autonomous work.** They tend to stop at "first plausible blocker" rather than push through. Cheap probes (30-min) are the sweet spot for sub-agents; multi-hour work is better driven directly.
