## 3. Maintainability findings

### 3.1 File / module concerns

- **`src/bake/v2.rs` is misnamed.** It emits v3. The file comment (`src/bake/v2.rs:1-6`) and the function name `bake_v2` say so explicitly: *"Despite the function name `bake_v2` — kept for source compatibility with the few internal callers — this module emits the v3 wire format"*. For a 0.2.0 hard fork, the source-compat reason evaporates. **Rename `src/bake/v2.rs` → `src/bake/composer.rs`** (or `src/bake.rs` since `mod.rs` is just 31 lines of re-exports), **rename `bake_v2` → `bake` / `bake_znpr`**, drop the apologetic comment.

- **`src/bake/mod.rs:1-23`** still describes the module as "ZNPR v2 byte-stream composer". Inconsistent with the v3 hard fork.

- **`src/lib.rs:121`** crate-level docstring closing line: *"the format (`ZNPR v2`) is the contract between them"*. Wrong post-fork.

- **`src/lib.rs:44`** doctest uses `bake_v2` in its example — public-API entrypoint should be renamed in lockstep with the symbol rename.

- **`src/model.rs:274`** `Model` struct doc says *"Parsed view over a ZNPR v2 model"*; `:298` `from_bytes` says *"Parse a v2 model from raw bytes"*. Both wrong post-fork.

- **`src/metadata.rs:1`** module comment says *"ZNPR v2 metadata blob"*. Update.

- **`src/error.rs:10-12`** `UnsupportedVersion` doc says *"v2 is the only version this crate parses"*. Now v3.

- **`src/bake/json.rs:1`** module title says *"JSON input schema for [`bake_v2`]"* and the chain of doc comments through this file says v2. Refresh.

- **README.md`** still references "ZNPR v2" in 4 places (lines 3, 15, 55, 57, 86). Hard-fork release means a README rewrite.

### 3.2 Magic numbers / hardcoded section offsets

`src/bake/v2.rs:321-330` defines `HEADER_SIZE`, `LAYER_ENTRY_SIZE`, and 7 `SECTION_OFF_*` constants. The same constants exist (with the same values) at `src/model.rs:111-112`. **Promote them to a single shared `model::constants` (or `model::wire`) module** that both the parser and composer import. Today, a section offset change requires editing two files (parser and composer) consistently or the bake silently mis-writes.

`src/inference.rs:131,160,189` hardcodes the SIMD chunk size `8` (output dim) inline. Fine, but worth a named `const FORWARD_CHUNK: usize = 8` per the docstring at `src/inference.rs:13` which calls out `f32x8` explicitly.

`src/output_spec.rs:107-110` — `Exp` transform `clamp(-30.0, 30.0)` — duplicated with `src/argmin.rs:513` and `output_spec.rs:111-122` (`Exp` clamp), all using the literal `-30.0, 30.0`. Promote `EXP_INPUT_CLAMP: (f32, f32) = (-30.0, 30.0)` so the units are consistent and the rationale (`f32::MAX ≈ 3.4e38`, `log(MAX) ≈ 88.7`) lives in one place.

### 3.3 `unsafe`

`#![forbid(unsafe_code)]` at `src/lib.rs:124` — verified. The crate is unsafe-free; `bytemuck` does the typed-slice reinterpret behind safe `try_cast_slice`. ✅

### 3.4 Public-API audit — `pub` vs `pub(crate)`

The pre-existing `docs/ZNPR_V3.md` lines 62-71 already flagged the yagni-trim. Concrete items needing visibility tightening:

- **`Header`, `LayerEntry`, `Section`, `LayerView`, `WeightStorage`** (`src/model.rs:165-272`) — every field is `pub`. Real consumers don't construct or pattern-match `Header`/`LayerEntry` directly; they go through `Model::n_inputs()` / `Model::layers()`. **Convert all fields to `pub(crate)`**, expose accessor methods. `WeightStorage` and `LayerView` are read by `zensim::mlp` for cross-validation against its Python trainer — those consumers DO pattern-match the variants. **Verify with zensim before locking down**; from the grep, zensim only reads `WeightStorage` variants and calls `WeightDtype::F32` etc. constructors, so the enum itself stays public; the **structs' fields** can shrink.

- **`inference::f16_bits_to_f32`** (`src/inference.rs:223`) is re-exported at `src/lib.rs:153` AND used by zensim's mlp code (one external call site found). Keep public — but document it as "stable for migration off the v2 reader; will be `pub(crate)` in 0.3+".

- **`output_spec::OutputTransform::from_byte`** is `pub(crate)` — correct.

- **`feature_transform::parse_feature_transforms`** is `pub(crate)` — correct.

- **`feature_transform::FeatureTransform::from_token`** is `pub(crate)` — correct; the public `as_token` companion at `:90` causes a rustdoc warning (`public documentation for as_token links to private item Self::from_token`). Either make `from_token` public (cheap, useful for round-trip tests) or drop the link.

- **`predictor::Predictor::predict_with_specs_transformed`** (`src/predictor.rs:211`) — 0 external usages. Currently `pub`. Move to `pub(crate)` or delete (see YAGNI).

- **Re-exports at `src/lib.rs:143-162`** — exports `bounds::FeatureBound, OutputBound, first_out_of_distribution, output_first_out_of_distribution`, `rescue::*`, `safety::*` — these all have **zero consumer usage** (verified by grep across zensim, zenwebp, zenjpeg, zenavif, zenjxl, zenpicker). See YAGNI verdicts.

### 3.5 Stringly-typed / primitive-soup APIs

- `inference::forward(model, features, scratch_a, scratch_b, output)` takes 4 raw slices. The `Predictor` already wraps this internally; `forward` is `pub` only because the module is `mod inference` (private) and the function isn't re-exported. **Verify it's not externally callable** — `lib.rs` doesn't re-export `forward`, so the `pub` on the function inside a private module makes it crate-internal. Fine as-is. Doc could be updated to clarify "crate-internal entry point; consumers go through `Predictor`".

- `argmin::pick_with_confidence` returns `Option<(usize, f32)>` — the `f32` is "gap to second-best." Returning a typed `Confidence { idx: usize, gap: f32 }` would clarify call sites (the gap is in score units, not zq points). 0 external usages today, so cheap to introduce. Pair with the `pick_with_confidence` keep/delete verdict.

- `Predictor::argmin_masked_in_range`, `argmin_masked_top_k_in_range`, `pick_with_confidence_in_range`, `argmin_masked_with_scorer_in_range`, `argmin_masked_top_k_with_scorer_in_range` (`src/predictor.rs:301-530`) — five methods that take `range: (usize, usize)` for the same purpose. A `Range<usize>` slice carved out by the caller would be idiomatic, OR an `OutputSlice { start: usize, len: usize }` wrapper. With 0 external usages today, this is the time to fix.

## 6. Test coverage gaps

Coverage is broad on the bake-and-parse round-trip side; thinner on the "real .bin loaded from disk" side.

- **No integration test loads a real shipped bake.** Every test bakes a fixture in-memory then parses it. The bakes shipped under `/mnt/v/output/zensim/synthetic-v2/.../v0_4*.bin` and `zenwebp/src/encoder/picker/zenwebp_picker_v0.1.bin` are exercised only by the codec crates. **Recommendation:** add a test that walks `tests/fixtures/*.bin` (committed real-but-tiny bakes) and runs `Model::from_bytes` + a `predict` call. Catches subtle wire drift between zentrain's Python emit and the Rust parse.
- **No benchmark.** Per the CLAUDE.md "zenbench not criterion" rule: this is a candidate for a `benches/predict.rs` that times `Predictor::predict` against an F16 228×64×1 model (zensim V0_4 shape) and an F16 51×32×12 model (zenjpeg picker shape). Today's "is the forward pass fast enough" question is unanswered.
- **Negative tests on malformed bytes:** present (BadMagic, version mismatch, dim mismatch, unknown activation/dtype tests cover the loader). Coverage looks complete.
- **No-default-features build is tested by `cargo build`; `cargo test --no-default-features` is not exercised in the local run.** CI may already run it — verify before publish.

---

**Bottom line:** the crate is in good shape. Tests are comprehensive, the runtime kernel is small and well-documented, and the v3 wire format is solid. The 0.2.0 fork is mostly **paperwork** (rename, strip v2 references, fix doc warnings) plus a **substantial public-API trim** — moving ~1000 LOC of speculative subsystems (safety, rescue, output_specs, top-k, scorer) behind cargo features so the lean runtime stays lean. Nothing is structurally wrong; this is the time to crystallize what the API actually delivers.
