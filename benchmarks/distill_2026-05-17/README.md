# Multi-codec distillation A/B — 2026-05-17

**Question:** can a per-codec MLP student, distilled from the joint
shared-trunk model, recover the joint training accuracy lift well
enough that we can ship distilled (no runtime change to zenpredict)?

**Answer: no — for codecs that matter, distillation regresses
significantly.** Ship the multi-codec inference path directly via the
new `Picker::predict_multi_codec` helper (parallel sub-agent built
this in `zenpredict`).

## Setup

3 codecs: zenjpeg + zenwebp + zenavif. Joint trunk: 96 → 48. Distill
student per codec: 192 × 192 × 192 leakyrelu MLP, 200 epochs, AdamW
+ cosine LR. Image-level 80/20 train/val split, seed `0xCAFE`.

Reproduce:

```bash
PYTHONPATH=<zenanalyze>/zentrain/examples:<zenanalyze>/zentrain/tools \
    python3 zentrain/tools/distill_multi_to_per_codec.py \
        --codec zenjpeg=examples/zenjpeg_picker_config.py:/home/lilith/work/zen/zenjpeg \
        --codec zenwebp=examples/zenwebp_picker_config.py:/home/lilith/work/zen/zenwebp \
        --codec zenavif=examples/zenavif_picker_config.py:/home/lilith/work/zen/zenavif \
        --out-dir benchmarks/distill_2026-05-17 \
        --hidden 96,48 --epochs 200 \
        --student-hidden 192,192,192 --student-epochs 200
```

## Held-out val argmin accuracy / mean overhead

| Codec | Teacher only | Joint direct | Distilled | Joint − Teacher | Distilled − Joint |
|---|---:|---:|---:|---:|---:|
| zenjpeg | 48.8% / 2.72% | 50.5% / 2.98% | 43.5% / 5.28% | **+1.7 pp** | **−7.0 pp** |
| zenwebp | 33.2% / 3.33% | 40.4% / 2.49% | 39.5% / 3.22% | **+7.2 pp** | **−0.9 pp** |
| zenavif | 25.6% / 4.59% | 23.6% / 4.02% | 20.8% / 4.99% | **−2.0 pp** | **−2.8 pp** |

## What the numbers say

1. **Joint training works for zenwebp (+7.2 pp acc, −0.84 pp overhead).**
   The codec gets cross-codec signal — confirms the +40 pp result Agent
   B reported (which was vs an older single-codec MLP, not vs the
   teacher-only baseline shown here).

2. **Joint training is roughly neutral on zenjpeg (+1.7 pp) and slightly
   negative on zenavif (−2.0 pp).** zenjpeg already has a strong
   teacher; the joint trunk doesn't add much. zenavif's hyperparam
   surface needs the variance-clamp fix Agent B's run noted (the
   3-way config gives zenavif's tiny output dimensionality
   disproportionate gradient weight).

3. **Distillation BREAKS zenjpeg.** −7.0 pp argmin acc + +2.30 pp
   mean overhead vs joint direct. The distilled student also loses
   to the teacher-only baseline. This is the same failure mode
   documented in `benchmarks/tabm_zenjpeg_2026-05-17.md`: **MSE on
   `bytes_log` doesn't optimize argmin selection.** Cell rankings
   that the joint model preserves internally get scrambled when a
   student tries to match the predicted `bytes_log` values in MSE.

4. **Distillation is roughly neutral for zenwebp (−0.9 pp) but
   asymmetric.** The codec with the largest joint training lift loses
   the least under distillation; the codec where joint training was
   only modestly useful loses the most. This is the worst possible
   shape — it means distillation can't reliably preserve the win.

## Decision

**Ship the multi-codec runtime helper.** The Rust agent built
`Picker::predict_multi_codec(codec_id, codec_features, size_class,
log_pixels, zq_norm)` in `zenpredict` (parallel sub-agent — see
`worktree-agent-a86642fbd4e3047a2`). The bake-side composer in
`zenpredict-bake` got a multi-codec schema section; ZNPR v3 encodes
the per-codec union-slot mapping so each codec just hands its
natural feature vector to the runtime.

Caveats / future work:

- **Train a single multi-codec bake** that ships once. The bake's
  ~3 MB per-codec JSON path was a stopgap; the right shape is one
  joint `.bin` that all codecs `include_bytes!` and call
  `Picker::predict_multi_codec(codec_id, …)` on. Pending the Rust
  agent's wiring + Python bake_picker.py update.

- **The MSE-on-bytes-log distillation failure mode is generic.** If
  a future workload needs distillation back to a single-codec MLP
  (e.g., for a runtime that can't be extended), the distill loss
  must be ranking-aware — e.g., listwise loss over the cell vector,
  or pair-margin loss that penalizes flipping the teacher's pairwise
  cell ordering. None of those are implemented in this prototype.

- **zenavif joint-vs-teacher is negative.** The 3-way trunk's
  variance normalization gives zenavif's tiny output dimensionality
  disproportionate gradient weight. Agent B's earlier writeup
  flagged a "clamp `loss_scale` floor" fix as queued; that's worth
  landing before we declare the multi-codec story stable.
