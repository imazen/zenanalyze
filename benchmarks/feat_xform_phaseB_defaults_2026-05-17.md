# Cross-codec transform-defaults analysis — Phase B (2026-05-17)

**Question:** for each feature used by multiple codec pickers, do the
codecs agree on the optimal input transform? If yes → push the
transform default into zenanalyze itself (the feature gets emitted
already-transformed). If no → keep transforms codec-specific in the
bake.

**Answer: no consensus.** Across 28 features used by ≥ 2 of the 3
codecs (zenjpeg / zenwebp / zenavif), **only 11 have the same winner
across the codecs that use them**; **29 features disagree**;
**0 features have 3-codec consensus**. The per-codec optimum is
irreducible. **zenanalyze should keep emitting raw values and let the
codec-specific bake apply its transform.**

## Method

Used the best per-codec screen result (best metric × vocabulary):

- zenjpeg → `singles+z_rmse` (9 single transforms, z-RMSE screen) — +3.74 pp argmin
- zenwebp → `v14+z_rmse` (14 transforms incl. runtime stacks) — **+24.54 pp argmin**
- zenavif → `v14+z_rmse` — **+2.57 pp argmin**

Per-feature winner from each codec's `screen_results.tsv` where
lift ≥ 0.005 and winner ≠ identity.

## 29 codec-disagreement features

Same feature gets different winners across codecs. Examples (full
list in the analysis output):

- `feat_colourfulness`: zenjpeg `clip_then_log1p`, zenwebp
  `clip_then_log1p_then_winsor`, zenavif `winsor_then_log` — **all
  three differ**
- `feat_aq_map_std`: jpeg `clip_then_log1p`, webp `winsor_then_log`,
  avif `clip_then_log1p` — webp diverges
- `feat_edge_slope_stdev`: jpeg `log`, webp `signed_cbrt`, avif
  `clip_then_log1p_then_winsor` — all three differ
- `feat_noise_floor_y_p50`: jpeg `winsor_p99`, webp `signed_cbrt`,
  avif `clip_then_log1p` — all three differ

These features distinguish by codec because each codec's per-cell
`bytes_log` distribution differs — same image-feature value maps to
different byte-cost ranges depending on which codec is encoding.

## Implications

### What this rules out

The earlier proposal (move `log` / `log1p` defaults into zenanalyze
for "universally-log'd" features) is **doubly superseded**:

1. The earlier negative finding: the production-supported runtime
   vocabulary has 14 transforms; plain `log` / `log1p` wins only
   ~5 % of the time anywhere. zenanalyze can't even ship the
   "universally log'd" defaults because those aren't the winners.

2. **This finding: the per-codec optimum is irreducible.** Even
   among the 11 features that have 2-codec consensus, no feature
   has 3-codec consensus. Pushing a transform into zenanalyze would
   force a one-size-fits-all on the 29 features where codecs disagree
   — measurably worse than letting the bake apply per-codec.

### What to do instead

**Keep zenanalyze emitting raw feature values.** The transform layer
lives in the bake artifact + runtime apply path, which is now
infrastructurally complete:

- `zenpredict::FeatureTransform` has all 14 runtime variants (singles
  + 5 stacks promoted in 0.2.1)
- `zenpredict-bake` validates transform params at bake time
- `train_hybrid._apply_feature_transform` and
  `bake_picker.py` forward all 14 tokens + per-feature params
- `feature_transform_sweep.py` auto-derives per-codec winners via
  Pearson / Spearman / z-RMSE screens

Each codec config declares its own `FEATURE_TRANSFORMS` +
`FEATURE_TRANSFORM_PARAMS` (auto-generated by the sweep). The
schema_hash gates compatibility — same image feature with different
codec-specific transforms is allowed because the bakes are codec-
specific.

### Output shaping (target side)

Not measured by these sweeps (the screen is INPUT-side only). The
picker's target is per-cell `bytes_log` already (natural log of bytes
emitted for that cell at that target_zq). Per-cell standardization
during distill is the production trainer's existing mechanism (see
`train_hybrid.py:2249` per-head normalization). No evidence that
changing output shaping helps.

**Future work** (not in this batch):
- Try `log1p(bytes)` vs `log(bytes)` as the target — tiny files with
  bytes ≤ ~50 are border cases. Probably negligible impact.
- For per-cell scalar heads (chroma_scale, lambda, effort), the
  existing `output_specs` mechanism already lets each codec declare
  bounds + sigmoid_scaled / round / etc. Out-of-distribution clipping
  is per-head. Already as flexible as needed.
- Loss objective: current is per-row MSE on standardized bytes_log;
  switching to listwise / pair-margin loss (mentioned in the TabM
  bench writeup as a way to optimize argmin directly) would change
  the screen's target distribution. Independent of input transforms;
  separate research question.

## Recommendation summary

1. **Do not change zenanalyze input defaults.** Raw values stay raw.
   Codec configs declare per-feature transforms.
2. **Do not push defaults into the trainer either.** No 3-codec
   consensus. Default `FEATURE_TRANSFORMS = {}` (current behavior)
   stays.
3. **Production codec configs adopt their sweep winners**:
   - zenjpeg → `benchmarks/feat_xform_zenjpeg_singles_zrmse_2026-05-17/`
   - zenwebp → `benchmarks/feat_xform_zenwebp_v14_2026-05-17/`
   - zenavif → `benchmarks/feat_xform_zenavif_v14_2026-05-17/`
4. **For new codecs**: run `feature_transform_sweep.py
   --codec-config <new> --screen-metric z_rmse --confirm` to auto-
   derive the codec's `FEATURE_TRANSFORMS` + `FEATURE_TRANSFORM_PARAMS`.
   Try Pearson and Spearman variants too; pick the metric whose
   `--confirm` delta is best. Stack-enabled vs singles-only is also
   per-codec — try both.
5. **Output shaping unchanged** for now. The per-cell standardization
   and `output_specs` mechanism are already as flexible as needed.
