# coefficient project audit, 2026-05-02

## What it is

**coefficient** is a distributed codec benchmarking framework designed
for large-scale image encoding sweeps. It's a Rust backend that runs
locally (single machine) or distributed on GCP Batch, encoding millions
of PNGs through codecs at multiple quality levels, computing perceptual
metrics (SSIMULACRA2, Butteraugli, zensim), and tracking complete
provenance (code version, git commit, config JSON) for every result.

Task-based worker model with no coordination bottleneck — workers pull
tasks from a Planner, write immutable records to a ResultStore (local
disk, GCS, or split local+cloud).

Deployment modes:
- Local filesystem for testing
- GCP Batch jobs with 1000s of parallel workers writing to GCS/Firestore
- Hybrid (split store: metadata local, blobs on GCS)

## Architecture

```
Sources (PNGs)
   │
   ▼
SourceRecord {hash, dims, corpus, bucket} ──► meta/sources/{hash}.json
                                              blobs/sources/{hash}.png
   │
   ▼
Planner generates EncodingTask × MetricTask
   │
   ▼
[Worker pool: rayon local OR GCP Batch with 1000+ tasks]
   │
   ▼
EncodingRecord {codec, version, commit, config_json, q, size, time, bytes_hash}
   │                              ▼
   │                              meta/encodings/{prefix}/{hash}/record.json
   │                              blobs/encodings/{prefix}/{hash}/data.{ext}
   ▼
MetricRecord {metric, version, score, time}
                             ▼
                             meta/metrics/{prefix}/{enc_hash}/{metric_id}.json

   │
   ▼
analysis/pareto_rows.csv         (raw training data — bytes + metrics per encoding)
analysis/source_features.json    (zenanalyze features per source)
analysis/selector_tree_rules.json (decision rules)
```

## Data plane

**Storage backends** (pluggable):
- `LocalStore` — filesystem only
- `GcsStore` — Google Cloud Storage end-to-end
- `SplitStore` — metadata local (fast lookups), blobs on GCS / R2 / DO Spaces

**Layout** (SplitStore):
```
<root>/
├── meta/
│   ├── sources/{hash}.json
│   ├── encodings/{prefix1}/{prefix2}/{full_hash}/record.json
│   └── metrics/{prefix1}/{prefix2}/{enc_hash}/{metric_id}.json
├── blobs/
│   ├── sources/{hash}.png
│   └── encodings/{prefix}/{hash}/data.{jpg|webp|avif|jxl|png}
└── analysis/
    ├── pareto_rows.csv
    ├── source_features.json
    └── selector_tree_rules.json
```

**Three core record types**:
- `SourceRecord` — sha256 of PNG, dimensions, corpus, bucket membership
- `EncodingRecord` — codec name + version + git_commit + config_json + quality + size + time + bytes_hash
- `MetricRecord` — metric name + version + score + compute_time

**Provenance is end-to-end**: every encoding record carries its codec
crate name, version, git commit, full config JSON.

**Determinism**: same `(source_hash, codec_config, quality, effort)` →
same task_id → automatic dedup if already encoded.

## Compute plane

**Local Worker** (`src/worker/mod.rs`):
- Reads tasks from a `TaskPlan` (generated by `Planner`)
- rayon-based, configurable encoding + metric thread pools (default 90% physical cores)
- Retry logic for metric tasks waiting on encoding completion
- Optional heartbeat file for watchdog

**Planner** (`src/planner/mod.rs`):
- Cartesian product of sources × codecs × quality levels
- `skip_completed` resumes after crashes
- Chunking for distributed execution (default chunk_size=1000)

**GCP Batch** (`src/cloud/batch.rs`, feature `batch`):
- N tasks submitted; each task runs `cloud_worker` binary
- No central queue — workers pull from local FS / GCS independently
- Job tracking via Firestore documents
- Stall detection prototype: Cloud Functions in `functions/cleanup/`
  cancel jobs stuck >15 min with zero progress

**Encoder version pinning**: Docker images (`Dockerfile.gpu`,
`Dockerfile`) bundle the exact Rust workspace. Cargo path deps to
sibling codecs (`mozjpeg-rs`, `butteraugli`, `turbo-metrics`,
zenjpeg/zenwebp/zenavif/zenjxl) come in at the build commit. Git tag /
commit captured in every EncodingRecord's provenance.

## Codec integration surface

**Trait** (`src/codec/mod.rs`):
```rust
trait Codec {
    fn name(&self) -> &str;
    fn encode(&self, rgb: &[u8], width: u32, height: u32, quality: u8) -> Result<EncodeResult>;
    fn provenance(&self) -> CodecProvenance; // crate name + version + commit + config
    fn extension(&self) -> &str;
    fn effort_range(&self) -> RangeInclusive<u8>;
}

struct EncodeResult {
    data: Vec<u8>,
    blob_hash: Hash,
    size_bytes: u64,
    encode_time_ms: f64,
}
```

**Registry pattern**:
```rust
let registry = CodecRegistry::with_efforts(&EffortConfig::standard());
// One codec instance per (family, subsampling, effort) combo
```

Codecs registered into `CodecRegistry`. Effort levels baked at
registry construction time.

**Existing integrations** (all 4 zen-codecs are in!):
- **zenjpeg** — Rust port of jpegli; 420/444/XYB variants
- **mozjpeg-rs** — Rust FFI; 420/444 variants
- **zenwebp** — pure Rust WebP
- **zenavif** — pure Rust AVIF
- **zenjxl** — pure Rust JPEG XL
- **zengif** — pure Rust GIF
- **CLI codec wrappers** (feature `cli-codecs`) — auto-detect:
  cjpegli, cjxl, cwebp, avifenc

## Top 3 questions zentrain needs answered before orchestration

1. **How to bridge coefficient outputs → zentrain inputs?**
   - coefficient emits `pareto_rows.csv` (54k rows, 17 cols with
     full `codec_config_json`).
   - zentrain `train_hybrid.py` expects two TSVs: `PARETO`
     (bytes/metric per image/config/q) and `FEATURES` (per-image
     `feat_*` columns).
   - Friction: coefficient's CSV has metrics already baked in (one
     row per encoding); zentrain expects features in a separate
     TSV with `feat_*` column names.
   - **Decision**: transform on-the-fly in zentrain (Python adapter)
     or have coefficient emit two separate TSVs from the start
     (matching zentrain's codec-config module contract)?

2. **How are codec configs represented end-to-end?**
   - coefficient stores full `config: serde_json::Value` per
     encoding.
   - zentrain `parse_config_name(name)` extracts categorical +
     scalar axes from a string config_name.
   - **Decision**: encode all knobs in config_name as tuples, or
     extract a curated subset? If coefficient produces 1000s of
     configs, does zentrain train on all or only Pareto-frontier?

3. **What's the SLA for "millions of images"?**
   - coefficient scales horizontally (GCP Batch with 1000+ tasks).
   - Per-worker overhead (Docker startup, GCS auth, heartbeat I/O)
     is non-trivial.
   - For 100-1000 image training, distributed isn't worth the
     overhead.
   - **Decision**: zentrain orchestrates coefficient locally, calls
     out to GCP Batch, or both with a mode switch?

## Friction with single-machine zentrain pipeline

- coefficient's pareto_rows.csv schema differs from zentrain's
  PARETO TSV schema. Manual bridging today.
- coefficient's `codec_config_json` is untyped; zentrain wants
  parsed categorical/scalar axes.
- Per-codec quality lists in coefficient (jpeg_qualities,
  avif_qualities, etc.) — zentrain expects a uniform target_zq grid.
- Two encoding sources (coefficient produces ground truth;
  zentrain runs its own per-codec sweep harnesses today). The
  zentrain harnesses should call coefficient instead.

## What would make zentrain ↔ coefficient orchestration cleaner

1. coefficient `--export zentrain` exporter — emits zentrain-shaped
   PARETO + FEATURES TSVs directly.
2. Typed config schema (Rust-side strong typing for `codec_config`)
   so zentrain doesn't need a stringly-typed regex parser.
3. Uniform target_zq grid across codecs (zentrain owns the grid;
   coefficient encodes against it).
4. Provenance threaded into picker.bin TLV metadata so a baked
   picker carries (corpus_hash, encoding_run_id, codec_commits,
   features_extractor_commit, train_seed).
